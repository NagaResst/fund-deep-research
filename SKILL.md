# 基金深度研究 Skill（完整版）

## 🎯 核心目标

输入基金代码 → **多步骤数据获取** → **强制完整性检查** → **联网搜索补充** → 生成高质量研究报告

**关键原则**：
1. ✅ 脚本是起点，不是终点
2. ✅ 数据缺失必须联网搜索补充
3. ✅ 报告不能有过多N/A字段
4. ✅ 所有数据必须是最新的

---

## 📋 执行流程（严格按顺序）

### Step 1：运行基础脚本获取初始数据

依次运行以下脚本，收集所有可用数据：

```bash
# 1. 基础信息
python skills/fund-deep-research/scripts/fetch_fund_basic.py <基金代码>

# 2. 风险指标（从天天基金直接获取）
python skills/fund-deep-research/scripts/fetch_risk_metrics.py <基金代码>

# 3. 持仓分析
python skills/fund-deep-research/scripts/analyze_holdings.py <基金代码>

# 4. 黑名单检查
python skills/fund-deep-research/scripts/check_blacklist.py <基金公司> <基金经理>

# 5. 政策匹配
python skills/fund-deep-research/scripts/policy_match.py <行业数据>
```

**重要**：保存所有JSON输出，用于后续检查。

### Step 2：🔍 数据完整性检查（强制执行）

**必须检查以下核心字段是否为N/A或空值**：

#### A. 基础信息（不能有任何N/A）
- [ ] 基金名称
- [ ] 基金类型（股票型/债券型/混合型）
- [ ] 成立时间
- [ ] 基金规模
- [ ] 基金经理姓名
- [ ] 基金公司名称

#### B. 业绩指标（至少要有近1年数据）
- [ ] 近1年收益率
- [ ] 夏普比率（或类似风险调整收益指标）
- [ ] 最大回撤

#### C. 持仓结构（至少要知道类型）
- [ ] 资产配置比例（股票/债券/现金）
- [ ] 前十大重仓（股票或债券）

**检查结果处理**：
- ✅ **全部有数据** → 跳到Step 4
- ⚠️ **部分缺失** → 进入Step 3联网搜索补充基础数据
- ❌ **大量缺失**（>50%）→ 先联网搜索基本信息，再继续

**第3轮：持仓结构补充**

#### 搜索工具使用规范

**必须使用 `search_web` 工具**，每次搜索后：
1. 读取搜索结果（使用 `fetch_content` 如果需要）

3. 记录数据来源和日期
4. 验证信息的时效性（优先选择最近3个月的信息）

#### 信息整合规则

- ✅ 多个来源一致 → 采用该数据
- ⚠️ 来源冲突 → 采用权威来源（天天基金、晨星、基金公司官网）
- ❌ 无法验证 → 标注"待确认"，不要编造

### Step 4：📊 数据验证与交叉核对

**在生成报告前，必须验证**：

1. **数据一致性检查**
   - 基金名称在各处是否一致
   - 净值数据是否合理（不会突然暴涨暴跌）
   - 规模数据是否在合理范围

2. **时效性检查**
   - 净值日期是否是最近的（T-1或当日）
   - 季报数据是否是最新的
   - 基金经理是否现任

3. **合理性检查**
   - 收益率是否合理（债券基金年化一般3%-8%，股票基金-30%到+50%）
   - 夏普比率是否合理（一般0-3之间）
   - 最大回撤是否符合基金类型

**发现问题时**：
- 重新搜索验证
- 标注数据疑点
- 在报告中说明

### Step 5：🔍 深度研究第4章所需数据（关键步骤·易忽略）

**重要提醒**：第4章“历史表现与市场/政策关联分析”需要**额外的深度联网搜索**，这部分工作容易被忽略！

**必须进行以下搜索**：

#### A. 全量季度数据获取（分析骨架）
```bash
# 运行季度表现脚本，获取基金成立以来所有季度的持仓变迁
python skills/fund-deep-research/scripts/analyze_quarterly_performance.py <基金代码>
```
*此脚本将输出该基金历史上所有的季度报告期、前五大重仓股及占比。AI 需以这些季度为单元进行逐一复盘。*

#### B. 逐季深度复盘：盈亏-持仓-通告三维绑定（核心工作）
对于脚本输出的**每一个历史季度**，必须执行以下深度对账流程：

1.  **获取官方通告原文（经理的“自白”）**：
    *   使用网络搜索：`"{基金名称} {年份}年第X季度报告 投资策略和运作分析"`。
    *   **目标**：提取经理对该季度操作的解释、对市场的看法以及对下一季度的展望。
2.  **绑定持仓变化（经理的“动作”）**：
    *   对比本季度与上一季度的前十大重仓股，识别：加仓、减仓、新进、退出。
    *   观察行业集中度变化：是更集中了还是更分散了？
3.  **关联季度盈亏与市场背景（市场的“反馈”）**：
    *   搜索该季度的净值增长率。
    *   搜索该季度的宏观大事、行业政策及市场风格（如：成长vs价值）。
4.  **逻辑一致性审计（AI 深度分析）**：
    *   **知行合一**：经理说看好 A，实际是否加仓了 A？
    *   **归因分析**：季度盈利是靠经理选股（Alpha），还是靠行业风口（Beta）？
    *   **逻辑演变**：经理的投资逻辑在过去几年是否发生了漂移？

#### B. 市场对比数据搜索
```bash
搜索关键词：
- "{基金名称} vs 沪深300 对比"（股票型）
- "{基金名称} vs 中债指数 对比"（债券型）
- "{基金名称} Alpha Beta 信息比率"
```

#### C. 重大政策事件影响搜索（⚠️ 必须精确到文号与量化目标）

**第1轮：历史政策文件精确搜索**
```
搜索关键词：
- "{持仓行业} 政策文件 文号 历年"
- "{持仓行业} 指导意见 补贴 退坡 通知 历年"
- "国家发改委 能源局 {持仓行业} 规划 指导意见"
- "发改能源 工信部 {持仓行业} {年份} 文件"
```

**第2轮：政策传导量化搜索**
```
搜索关键词：
- "{基金名称} 政策发布后 净值 涨跌"
- "{持仓行业} 政策红利 股价 反应"
- "{基金代码} 政策敏感度 Beta"
```

**第3轮：最近6个月新政搜索（必须执行）**
```
搜索关键词：
- "{持仓行业} 2026年 最新政策 利好 补贴"
- "工信部 发改委 {持仓行业} 2026 通知 意见"
- "{持仓行业} 省级 地方 补贴 规划 2026"
- "{持仓行业} 供给侧 反内卷 竞争规范 2026"
```

**搜索后必须整理**：
- 每条政策记录：**文号 / 发布部门 / 发布日期 / 量化目标 / 补贴金额或装机目标**
- 按政策周期分类：**红利期 / 调整期 / 修复期**，标注每段区间的净值涨跌幅
- 填写政策对照表（详见Section 4C要求）

#### D. 市场周期表现搜索
```bash
搜索关键词：
- "{基金名称} 牛市 熊市 表现"
- "{基金名称} 不同市场环境 适应性"
- "{基金经理姓名} 投资风格 择时能力"
```

**搜索要求**：
- 至少进行4轮搜索（A/B/C/D各一轮）
- 每轮搜索后使用`fetch_content`读取详细内容
- 提取关键时间节点、涨跌幅、驱动因素
- 记录数据来源和日期

**注意**：此步骤是生成高质量报告的关键，绝不能跳过！

### Step 6：📜 生成基金发展编年史（新增·核心能力）

**目标**：将基金从成立至今的完整历程串联起来，形成有深度的叙事。

**必须执行以下步骤**：

#### A. 收集完整时间线数据
运行脚本获取趋势分段结果：
```bash
python skills/fund-deep-research/scripts/trend_segmentation.py <基金代码>
```

#### B. 逐段归因与言行一致性审计（AI 核心工作）
对于脚本输出的**每一个波段**或**每一个季度**，执行以下深度对账：

1. **搜索官方通告原文**：
   - 使用网络搜索关键词：`"{基金名称} {年份}年{季度}报告 投资策略和运作分析"`。
   - **目标**：提取经理在当时的原始文字表述（如“看好上游资源”、“逐步减仓”等）。
2. **对比持仓动作**：
   - 结合 `analyze_quarterly_performance.py` 的输出，看经理是否真的按说的做了。
   - **识别风险**：如果经理说“防守”，但持仓反而更集中了，必须标注为“言行不一”。
3. **输出小节**：为每个波段生成复盘，包含：时间段、净值表现、官方观点、实际持仓、深度点评。

#### C. 历任经理深度追踪与风格演变审计
如果基金发生过经理变更，必须将分析颗粒度细化到**“人”**：

1.  **交接期动作还原**：
    *   对比新旧经理交替前后两个季度的**重仓股重合度**。是新经理“萧规曹随”，还是“推倒重来”？
    *   观察换手率变化：新经理上任是否伴随剧烈的调仓？
2.  **投资观念演变史**：
    *   **纵向对比**：搜索该经理在不同年份（如牛市顶点 vs 熊市底部）的官方通告。
    *   **核心追问**：他的投资逻辑是自洽且稳定的，还是随市场热点频繁摇摆的？
3.  **长期言行一致性画像**：
    *   不要只看某一季度，要总结他在**过去 3-5 年**的整体表现。
    *   **识别惯性**：他是否有“口头看好但实际不买”或“高位喊话接盘”的历史劣迹？

#### D. 机构风险扫描（新增·一票否决项）
运行 `scan_institutional_risk.py` 获取搜索关键词，并利用 `search_web` 逐一排查：
- **合规红线**：近 3 年是否有证监会处罚或监管函？
- **舆情监控**：是否有内幕交易传闻、维权事件或负面热搜？
- **管理疲劳**：经理在管产品是否超过 10 只或规模超 500 亿？

### Step 7：📝 AI直接生成完整研究报告（禁止使用脚本）

**重要**：报告必须由AI直接生成，**严禁使用任何脚本自动生成报告**！

**报告生成要求**：
1. ✅ 基于Step 1-5收集的所有数据
2. ✅ 结合联网搜索补充的信息
3. ✅ AI进行深度分析和综合判断
4. ✅ 确保报告质量达到专业水准
5. ❌ **禁止使用generate_report.py等脚本**

**报告必须包含以下章节**（缺一不可）：

#### 1. 基础信息表
- 基金代码、名称、类型
- 成立时间、规模
- 基金经理、公司
- 费率结构

#### 2. 业绩表现
- 近1月、3月、6月、1年、3年收益率
- 与同类平均对比
- 排名情况

#### 3. 风险评估
- 夏普比率
- 最大回撤
- 波动率
- 风险等级

#### 4. 历史表现与市场/政策关联分析（核心章节·深度复盘·全量趋势覆盖）

**必须深入分析基金历史涨跌与市场环境、政策变化的关联性**。本章节不再限制事件数量，而是**严格遵循净值曲线的每一个显著波段**进行全量复盘。

##### A. 全量趋势分段复盘（基于脚本输出）
**根据 `trend_segmentation.py` 输出的每一个波段，逐一进行分析**：

对于每一个显著波段（涨跌幅 > 5%），必须包含：
1. **时间段与表现**：起止日期、累计涨跌幅、持续时间。
2. **市场环境定性**：是普涨牛市、结构性行情、系统性熊市还是震荡市？
3. **关键驱动因素**：
   - **政策因素**：是否有国家级/行业级重大政策发布？（必须精确到文号）
   - **市场因素**：流动性变化、外围市场冲击、风格切换。
   - **个股/行业因素**：重仓股暴雷或爆发、行业景气度突变。
4. **基金经理应对**：
   - **调仓动作**：加仓/减仓/换股/躺平（基于季报持仓变化推断）。
   - **观点验证**：经理在季报中说的 vs 实际做的。
5. **事后评价**：这次操作是神来之笔还是致命失误？

**注意**：如果连续多个波段属于同一个宏观大背景（如“双碳政策红利期”），可合并为一个大章节叙述，但内部需保留各小波段的详细数据。

##### B. 与市场指数的对比分析
**将基金表现与相关指数进行对比**：

1. **基准指数选择**
   - 股票型基金：沪深300、中证500、创业板指
   - 债券型基金：中债综合指数、10年期国债收益率
   - 混合型基金：偏股混合型基金指数
   - 行业主题基金：对应行业指数（如新能源指数、半导体指数）

2. **超额收益来源分解**
   - **择时能力**：基金经理是否在市场低点加仓、高点减仓？
   - **选股能力**：重仓股是否跑赢行业平均？
   - **行业配置**：行业轮动是否成功？
   - **风格因子**：价值/成长、大盘/小盘风格暴露

3. **不同市场环境下的表现**
   - **牛市期间**：是否跟上或超越指数？
   - **熊市期间**：是否有效控制回撤？
   - **震荡市期间**：是否有稳定的超额收益？
   - **风格切换期间**：是否及时调整风格？

**量化指标**：
- Beta值（相对于基准指数的敏感度）
- Alpha值（超额收益）
- 信息比率（风险调整后超额收益）
- 跟踪误差（与基准指数的偏离度）

##### C. 重大政策事件影响分析（⚠️ 必须精确到文号、量化目标、净值传导幅度）

**【政策周期分段框架】**

首先将基金历史划分为若干「政策周期段」，每段必须包含：

| 字段 | 要求 |
|------|------|
| 阶段名称 | 红利期 / 调整期 / 修复期 / 政策真空期 |
| 时间区间 | XXXX-MM 至 XXXX-MM |
| 核心政策 | 政策全称 + 文号（如发改能源〔2017〕1701号） |
| 政策量化目标 | 如"到2025年新型储能装机≥3000万千瓦" |
| 区间净值涨跌幅 | XX% |
| 同期沪深300 | XX% |
| 超额收益 | XX% |

**必须输出「政策-净值对照量化表」**（每个完整周期一行）：

| 时间段 | 主要政策（含文号） | 政策核心量化目标 | 基金净值增长率 | 同期沪深300 | 超额收益 |
|--------|-------------------|-----------------|--------------|------------|----------|
| 2017年 | 《储能指导意见》发改能源〔2017〕1701号 | 十三五期间储能商业化初期 | XX% | XX% | XX% |
| ... | ... | ... | ... | ... | ... |

**1. 正面政策案例（每个至少包含以下字段）**
- **政策全称 + 文号 + 发布部门 + 日期**
- **政策核心量化目标**（装机目标/补贴金额/渗透率目标等）
- **对持仓行业的传导路径**：政策→行业需求/供给→重仓股利润→基金净值
- **净值反应量化**：政策发布后 1个月 / 3个月 / 6个月 的涨跌幅
- **基金经理应对**：是否在政策发布前后调整仓位

**2. 负面政策案例（每个至少包含以下字段）**
- **政策全称 + 文号 + 发布部门 + 日期**
- **负面内容摘要**（补贴退坡幅度/监管要求/禁止事项等）
- **对持仓的冲击路径**
- **净值反应量化**：政策发布后各阶段跌幅
- **基金经理是否及时减仓**（通过持仓变化判断）

**3. 最近6个月最新政策专项分析（必须单独成段）**
- 列出最近6个月内发布的所有相关政策
- 按以下维度分类：供给侧（规范竞争/质量监管）/ 需求侧（补贴/目标/采购）/ 技术侧（研发补贴/首台套奖励）
- 分析每条政策对基金重仓股的直接影响
- 给出「政策红利传导路径」：政策→受益细分领域→基金重仓股→预期净值弹性

**4. 政策敏感度评估**
- 敏感程度（高/中/低）+ 量化依据（政策发布后平均反应幅度）
- 政策传导时间滞后（即时/数周/数月），并举例说明
- 历史政策驱动型涨跌占总涨跌幅的比例估算

**搜索关键词示例**：
- "{持仓行业} 历年政策变化 影响分析"
- "{基金名称} 政策敏感度"
- "中国 {年份} 重大经济政策 对市场影响"

##### D. 市场周期适应性评估
**评估基金在不同市场周期中的适应能力**：

1. **经济周期**
   - 复苏期表现（通常利好周期股、金融股）
   - 过热期表现（通常利好资源股）
   - 滞胀期表现（通常利好防御性板块）
   - 衰退期表现（通常利好债券、现金）

2. **流动性周期**
   - 宽松期表现（利好成长股、高估值股票）
   - 收紧期表现（利好价值股、低估值股票）

3. **风险偏好周期**
   - 风险偏好上升期（利好小盘股、题材股）
   - 风险偏好下降期（利好大盘股、蓝筹股）

**评估结论**：
- 该基金最适合的市场环境是什么？
- 在哪些市场环境下表现较差？
- 基金经理是否有逆周期操作能力？

##### E. 归因分析总结
**综合以上分析，给出归因结论**：

1. **历史收益的主要来源**
   - XX%来自市场Beta（跟随大盘）
   - XX%来自行业配置Alpha
   - XX%来自个股选择Alpha
   - XX%来自择时操作

2. **基金经理的核心能力**
   - 擅长什么？（选股/择时/行业轮动/风险控制）
   - 不擅长什么？（需要规避的市场环境）

3. **可复制性评估**
   - 历史业绩有多少可以归因于运气（市场红利）？
   - 有多少可以归因于能力（可持续的Alpha）？
   - 未来能否复制历史业绩？

## �️ 结构化数据转换（Web 平台 JSON）

研究报告生成后，可选择将其转换为 Web 平台使用的结构化 JSON 数据，用于在线展示。

### 目标文件路径

```
web-platform/public/data/{基金代码}.json   ← 基金详情数据
web-platform/public/data/index.json        ← 基金清单（需同步更新）
```

---

### JSON Schema（14个顶层字段）

```
id, meta, basic, fees, scale, performance, risk,
holdings, managers, policy, exclusionCheck, scoring, navHistory, tracking
```

每个字段的关键子字段：

| 字段 | 关键内容 |
|------|---------|
| `id` | 基金代码字符串 |
| `meta` | reportDate, dataDate, disclaimer |
| `basic` | code, fullName, shortName, type, subType, riskLevel, riskCode(1-5), foundDate, manager(公司), navFallback, inceptionReturn, companyShort |
| `fees` | management, custodian, salesService, subscriptionMax, breakdown数组 |
| `scale` | nav(亿元), shares(亿份), date |
| `performance` | stages数组, annual数组, quarterly数组（每条含fund/peer/hs300/rank/rankTotal/quartile） |
| `risk` | volatility, sharpe, maxDrawdown, radarDimensions, riskBreakdown, positionHistory |
| `holdings` | date, stockRatio, top10数组, sectors数组, bondStructure(债基), policyLinks |
| `managers` | current对象（含historicalFunds数组, managerId）, history数组 |
| `policy` | tags, scenarios(悲观/基准/乐观), policyBreakdown, fifteenFive, longTermRisks, dualTimeline, timeline |
| `exclusionCheck` | 数组，每条含 item, pass(bool), note |
| `scoring` | total(0-100), dimensions数组, recommendation, recommendationType(buy/hold/cautious/sell), reasoning, conclusion, suitableFor, allocationSuggestions |
| `navHistory` | 通常为空数组 [] （由 API 实时填充） |
| `tracking` | weekly数组, quarterly数组, alerts数组（含level: critical/warning/info） |

---

### 转换流程

#### Step A：从研究报告提取数据

逐章读取 `.md` 研究报告，按 Schema 字段映射：

- **第一章（基础信息）** → `basic`, `fees`, `scale`
- **第二章（业绩表现）** → `performance.stages`, `performance.annual`, `performance.quarterly`
- **第三章（风险评估）** → `risk`（volatility/sharpe/maxDrawdown）
- **第四章（历史与政策）** → `policy.timeline`, `policy.dualTimeline`
- **第五章（持仓结构）** → `holdings`
- **第六章（基金经理）** → `managers`
- **第七章（政策预测）** → `policy.tags`, `policy.scenarios`, `policy.policyBreakdown`, `policy.fifteenFive`, `policy.longTermRisks`
- **第八章（排除法检查）** → `exclusionCheck`（10条，每条 pass: true/false）
- **第九章（投资建议）** → `scoring`
- **第十章（跟踪计划）** → `tracking`

#### Step B：填写关键字段时的注意事项

**riskCode 对照**：
- R1 = 1（低风险）
- R2 = 2（中低风险）
- R3 = 3（中风险）
- R4 = 4（中高风险）
- R5 = 5（高风险）

**recommendationType 枚举**：
- `"buy"` — 推荐/强烈推荐
- `"hold"` — 持有/观望
- `"cautious"` — 谨慎
- `"sell"` — 减持/放弃

**dualTimeline 颜色规则**（type 字段）：
- manager 端：`positive`=红, `highlight`=黄, `warning`=绿(警示), `neutral`=灰
- market 端：bull/policy=`bull`, neutral=`neutral`, bear=`bear`

**null 的使用原则**：
- 数据在报告中明确缺失（如未公开） → 填 `null`
- 不要凭估算填写数字，除非报告有明确说明

**债券型基金特殊字段**：
- `holdings.bondStructure` 数组（政策性金融债/信用债/可转债等分类）
- `basic.riskCode` 通常为 2（中低风险）
- 行业集中度检查项 `exclusionCheck` 通常 pass: true

**ETF/指数基金**：
- `holdings.top10` 的 ratio 若未在报告中逐一列出可全部填 `null`
- 需填写 `tracking`（trackingIndex, trackingError, correspondingETF 等）

**字符串中的引号**：
- JSON 字符串值内不使用英文双引号，改用中文书名号 `「」`

**exclusionCheck 行业集中度**：
- 行业集中度 > 70% → `pass: false`（指数基金也标注，note 中说明为指数设计特性）

#### Step C：填写 managerId

基金经理的 managerId 从东方财富页面获取：

```bash
# 从基金f10页面抓取
curl -s "https://fundf10.eastmoney.com/jjjl_{基金代码}.html" | grep -oE 'manager/[0-9]+\.html' | head -3
```

将得到的数字 ID 填入 `managers.current.managerId`（字符串格式）。

多经理基金填主要经理的 ID；填入后前端将自动从东方财富实时拉取该经理名下所有在管基金数据。

### 更新 index.json

`index.json` 是首页基金清单，每新增一只基金需同步追加一条记录：

```json
{
  "id": "基金代码",
  "shortName": "基金简称",
  "type": "股票型",
  "subType": "指数型-股票",
  "riskLevel": "中高风险",
  "riskCode": 4,
  "return1Y": 54.83,
  "return3Y": null,
  "score": 82,
  "recommendation": "推荐，科技主题核心配置",

  "navFallback": 1.5253,

  "manager": "金泽宇"

```

**字段来源**：
- `return1Y` / `return3Y` → 研究报告第二章业绩数据
